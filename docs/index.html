Speech-to-Text (STT) Tests

Test 1.1 — Clean Speech Accuracy

Baseline WER/CER across diverse English datasets ?
Understanding This Test

This is the most fundamental STT test: how accurately does the model convert spoken words to text, using studio-quality recordings with known transcripts?

WER (Word Error Rate)
The percentage of words the model gets wrong. It counts substitutions ("cat" → "cap"), deletions (missed words), and insertions (extra words). Lower is better; human-level is roughly 5%.
CER (Character Error Rate)
Same idea but at the character level. Catches minor misspellings that WER might over-penalize.
SER (Sentence Error Rate)
The fraction of sentences with at least one error. A high SER means errors are spread across many utterances, not concentrated in a few.
Bootstrap 95% CI
A statistical confidence interval. We resample the results 1,000 times and report the range where the true WER likely falls. Narrower = more certain.
jiwer
The open-source Python library used to compute all these metrics. It normalizes text (lowercase, strip punctuation) before comparing.

Multiple datasets are tested because each has different speakers, accents, recording conditions, and topics — revealing where the model is strong or weak.

Test 1.2 — Batch Performance

Real-Time Factor & latency by audio duration ?
Understanding This Test

Measures how fast the model transcribes audio when given a complete file at once (as opposed to streaming chunks in real time).

RTF (Real-Time Factor)
Processing time ÷ audio duration. An RTF of 0.01 means 1 second of audio is transcribed in just 0.01 seconds. Lower is better.
RTFx
The inverse of RTF, shown as a multiplier. "150x" means the model processes audio 150 times faster than playback speed.
P50 / P90 / P99
Percentile latencies. P50 = median (half of requests are faster). P90 = 90th percentile. P99 = 99th percentile, catching rare slow outliers. A big gap between P50 and P99 means unpredictable performance.
Concurrency
Multiple requests sent simultaneously, simulating multiple users. Tests whether the server maintains speed under parallel load or slows down.

This test matters for production capacity planning: can the server handle your expected throughput?

Test 1.3 — Streaming Performance

TTFW, finalization latency, stability ?
Understanding This Test

In streaming mode, audio is sent in small 100 ms chunks — just like a live microphone. The model returns partial transcriptions that update in real time, then a final confirmed result.

TTFW (Time to First Word)
How long after audio starts sending until the first transcribed word appears on screen. Critical for live captioning — users notice delays above ~500 ms.
Finalization Latency
The delay between sending the last audio chunk and receiving the final, confirmed transcript. Lower means users see the "settled" result sooner.
Streaming RTF
Must be ≤ 1.0 to keep up with real-time audio. A value of 1.02 means the system is barely keeping pace and may lag behind on longer recordings.
Stability Rate
How often partial results remain consistent as more audio arrives. Low stability means earlier words "jump" to different words as context changes — visually distracting for users.

Streaming vs Batch WER is compared to quantify the accuracy cost of real-time processing. Streaming is typically slightly worse because the model has less context for early words.

Test 1.4 — REST vs gRPC

Interface latency comparison ?
Understanding This Test

The same STT model is accessible via two different communication protocols. This test sends identical audio through both to compare speed.

gRPC
A high-performance binary protocol by Google using HTTP/2. Supports native streaming, persistent connections, and efficient serialization. Preferred for server-to-server communication.
REST
The standard HTTP/JSON approach used by most web APIs. Easier to integrate (works in browsers, behind proxies) but adds overhead from JSON encoding.
SAP AI Core Proxy
The Riva gRPC server sits behind SAP AI Core's HTTP gateway, which adds a small translation layer for REST calls.

Why TTFW (427 ms) looks larger than Streaming Final (141 ms) and REST Total (158 ms): These three rows measure completely different things and should not be compared directly.

Streaming TTFW — 427 ms
Measured from stream open to the moment the first partial word appears. The model must receive several 100 ms audio chunks before it has enough context to predict anything. ~4 chunks in, the first word shows up.
Streaming Final — 141 ms
Measured from the last audio chunk sent to the final confirmed transcript. By this point the model has already heard everything — it just settles the last few words. This is only the tail-end latency, not total time.
REST Total — 158 ms
REST does not stream. The entire 30-second file is sent in one request and one response is returned. This 158 ms is a complete batch round-trip on a different interface — an independent measurement, not part of the gRPC streaming timeline.

The gRPC streaming timeline looks like: stream open → first word at ~427 ms → audio ends at ~30,000 ms → final result at ~30,141 ms. REST's 158 ms is a separate, parallel measurement.

Test 1.5 — Noise Robustness

WER degradation under additive noise ?
Understanding This Test

Real-world audio is rarely studio-clean. This test mixes clean speech with different types of noise at various volumes to see how accuracy degrades.

SNR (Signal-to-Noise Ratio)
Measured in decibels (dB). Higher = cleaner audio. SNR 20 dB = quiet office. SNR 0 dB = noise is as loud as speech. SNR −5 dB = noise is louder than speech.
Gaussian Noise
Random static/hiss — a synthetic worst-case scenario. Generated mathematically, not from real recordings.
Babble Noise
Overlapping human voices, simulating a crowded room or open-plan office. From the DEMAND dataset.
Traffic / Office Noise
Real-world environmental recordings from the DEMAND dataset — road sounds, keyboard clicks, HVAC, phone rings.
DEMAND Dataset
A public collection of real-world noise recordings from 15 environments, widely used in speech research.

The results tell you the minimum recording quality (SNR level) needed for usable transcription with this model.

Why Vietnamese and Spanish score higher
Speakers whose first language is Vietnamese or Spanish use phonological patterns (tones, vowel qualities, consonant clusters) that differ substantially from English. This causes pronunciation deviations the model wasn't trained to handle, raising WER.

Test 1.7 — Long-Form Audio

Hallucination, repetition, and accuracy on long audio ?
Understanding This Test

Most STT benchmarks use short clips (5–30 s). This test checks what happens with recordings lasting 40–90 seconds or more — closer to real meetings, lectures, and podcasts.

Hallucination
When the model outputs words or phrases that were never spoken. Measured by checking for 5-word sequences in the transcript that don't appear anywhere in the reference text.
Repetition
When the model gets stuck in a loop, repeating the same phrase over and over — a known failure mode in neural sequence models.
Concatenated Audio
Multiple short clips joined end-to-end to create longer files. Tests whether the model handles extended context or degrades after a certain length.
CTC (Connectionist Temporal Classification)
The decoding method used by Parakeet. CTC models process audio frame-by-frame without explicit attention, which can limit their effective context window on very long audio.

If the model degrades past ~40 seconds, production systems should chunk long audio into shorter segments — adding complexity but preserving accuracy.

Test 1.9 — Output Quality

Punctuation, capitalisation, number formatting ?
Understanding This Test

Beyond getting words right, can the model produce "ready to read" text — with proper punctuation, capitalization, and number formatting?

Punctuation F1
A balanced score (0–100%) combining precision (are the inserted punctuation marks correct?) and recall (are the required marks present?). Scored separately for periods, commas, question marks, and exclamation points.
Capitalisation Accuracy
Whether sentence-initial words and proper nouns are correctly uppercased.
Number Formatting
Whether spoken numbers become digits or symbols ("$5 million") rather than being spelled out ("five million dollars").

This matters for downstream use: meeting minutes, captions, and documents all need formatted text. Models that produce raw lowercase text without punctuation require expensive post-processing pipelines.

Test 2.5 — Streaming TTFB & RTF

Latency by text length and interface ?
Understanding This Test

How quickly does the TTS model start producing audio after receiving text? This is the single most important metric for conversational AI and voice assistants.

TTFB P50 (Time to First Byte, median)
How long after sending text the first audio chunk arrives — i.e. how soon the voice starts speaking. P50 = median across 50 requests. Under 200 ms feels instantaneous; over 500 ms feels laggy. In a voice assistant this is the audible pause before the first syllable.
RTF P50 (Real-Time Factor, median)
synthesis_time ÷ audio_duration. RTF 0.10 means audio is generated 10× faster than real-time. Must be <1.0 for streaming to work — otherwise audio cannot arrive fast enough to play continuously. Lower is better.
Total P50 (End-to-end latency, median)
Wall-clock time from sending text to receiving the last audio chunk. Equals TTFB + full streaming time for the rest of the audio, so it grows linearly with text length. Relevant for batch use cases where the whole file is needed before playback; for conversational AI, TTFB is what matters.
P50 / P90 / P99
Percentile latencies. P50 = median (half of requests are faster). P90 = 90th percentile. A large gap between P50 and P90 means occasional slow outliers.
Text Length Buckets
Requests grouped by input length (10, 25, 75, 150 words). TTFB should stay roughly constant across buckets — a good streaming system starts outputting audio before finishing the full synthesis, so longer text does not delay the first byte.

gRPC uses native streaming (audio chunks arrive as generated). REST typically buffers more before responding, hence the ~30 ms TTFB gap between the two interfaces.

A measure of how alike two voice fingerprints are. It ignores absolute magnitude and measures the angle between two vectors — 1.0 means they point in the same direction (same voice), 0.0 means no relationship.
