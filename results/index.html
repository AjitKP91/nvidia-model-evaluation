Speech-to-Text (STT) Tests

Test 1.1 — Clean Speech Accuracy

Baseline WER/CER across diverse English datasets ?
Understanding This Test

This is the most fundamental STT test: how accurately does the model convert spoken words to text, using studio-quality recordings with known transcripts?

WER (Word Error Rate)
The percentage of words the model gets wrong. It counts substitutions ("cat" → "cap"), deletions (missed words), and insertions (extra words). Lower is better; human-level is roughly 5%.
CER (Character Error Rate)
Same idea but at the character level. Catches minor misspellings that WER might over-penalize.
SER (Sentence Error Rate)
The fraction of sentences with at least one error. A high SER means errors are spread across many utterances, not concentrated in a few.
Bootstrap 95% CI
A statistical confidence interval. We resample the results 1,000 times and report the range where the true WER likely falls. Narrower = more certain.
jiwer
The open-source Python library used to compute all these metrics. It normalizes text (lowercase, strip punctuation) before comparing.

Multiple datasets are tested because each has different speakers, accents, recording conditions, and topics — revealing where the model is strong or weak.

View source →

Test 1.2 — Batch Performance

Real-Time Factor & latency by audio duration ?
Understanding This Test

Measures how fast the model transcribes audio when given a complete file at once (as opposed to streaming chunks in real time).

RTF (Real-Time Factor)
Processing time ÷ audio duration. An RTF of 0.01 means 1 second of audio is transcribed in just 0.01 seconds. Lower is better.
RTFx
The inverse of RTF, shown as a multiplier. "150x" means the model processes audio 150 times faster than playback speed.
P50 / P90 / P99
Percentile latencies. P50 = median (half of requests are faster). P90 = 90th percentile. P99 = 99th percentile, catching rare slow outliers. A big gap between P50 and P99 means unpredictable performance.
Concurrency
Multiple requests sent simultaneously, simulating multiple users. Tests whether the server maintains speed under parallel load or slows down.

This test matters for production capacity planning: can the server handle your expected throughput?

Test 1.3 — Streaming Performance

TTFW, finalization latency, stability ?
Understanding This Test

In streaming mode, audio is sent in small 100 ms chunks — just like a live microphone. The model returns partial transcriptions that update in real time, then a final confirmed result.

TTFW (Time to First Word)
How long after audio starts sending until the first transcribed word appears on screen. Critical for live captioning — users notice delays above ~500 ms.
Finalization Latency
The delay between sending the last audio chunk and receiving the final, confirmed transcript. Lower means users see the "settled" result sooner.
Streaming RTF
Must be ≤ 1.0 to keep up with real-time audio. A value of 1.02 means the system is barely keeping pace and may lag behind on longer recordings.
Stability Rate
How often partial results remain consistent as more audio arrives. Low stability means earlier words "jump" to different words as context changes — visually distracting for users.

Streaming vs Batch WER is compared to quantify the accuracy cost of real-time processing. Streaming is typically slightly worse because the model has less context for early words.

Test 1.4 — REST vs gRPC

Interface latency comparison ?
Understanding This Test

The same STT model is accessible via two different communication protocols. This test sends identical audio through both to compare speed.

gRPC
A high-performance binary protocol by Google using HTTP/2. Supports native streaming, persistent connections, and efficient serialization. Preferred for server-to-server communication.
REST
The standard HTTP/JSON approach used by most web APIs. Easier to integrate (works in browsers, behind proxies) but adds overhead from JSON encoding.
SAP AI Core Proxy
The Riva gRPC server sits behind SAP AI Core's HTTP gateway, which adds a small translation layer for REST calls.

Why TTFW (427 ms) looks larger than Streaming Final (141 ms) and REST Total (158 ms): These three rows measure completely different things and should not be compared directly.

Streaming TTFW — 427 ms
Measured from stream open to the moment the first partial word appears. The model must receive several 100 ms audio chunks before it has enough context to predict anything. ~4 chunks in, the first word shows up.
Streaming Final — 141 ms
Measured from the last audio chunk sent to the final confirmed transcript. By this point the model has already heard everything — it just settles the last few words. This is only the tail-end latency, not total time.
REST Total — 158 ms
REST does not stream. The entire 30-second file is sent in one request and one response is returned. This 158 ms is a complete batch round-trip on a different interface — an independent measurement, not part of the gRPC streaming timeline.

The gRPC streaming timeline looks like: stream open → first word at ~427 ms → audio ends at ~30,000 ms → final result at ~30,141 ms. REST's 158 ms is a separate, parallel measurement.

Test 1.5 — Noise Robustness

WER degradation under additive noise ?
Understanding This Test

Real-world audio is rarely studio-clean. This test mixes clean speech with different types of noise at various volumes to see how accuracy degrades.

SNR (Signal-to-Noise Ratio)
Measured in decibels (dB). Higher = cleaner audio. SNR 20 dB = quiet office. SNR 0 dB = noise is as loud as speech. SNR −5 dB = noise is louder than speech.
Gaussian Noise
Random static/hiss — a synthetic worst-case scenario. Generated mathematically, not from real recordings.
Babble Noise
Overlapping human voices, simulating a crowded room or open-plan office. From the DEMAND dataset.
Traffic / Office Noise
Real-world environmental recordings from the DEMAND dataset — road sounds, keyboard clicks, HVAC, phone rings.
DEMAND Dataset
A public collection of real-world noise recordings from 15 environments, widely used in speech research.

The results tell you the minimum recording quality (SNR level) needed for usable transcription with this model.

Test 1.6 — Accent Robustness

WER by accent group ?
Understanding This Test

Evaluates whether the model performs equally well across different English accents and non-native speakers — a fairness and reliability check.

English Accent Dataset (HuggingFace)
A publicly available dataset with 23 distinct English accents (US, British, Indian, Australian, Canadian, Irish, Scottish, South African, and more), sourced from VCTK, EDACC, and VoxPopuli corpora. Each row has a transcript and an accent label — no authentication required.
WER per Accent
The model is evaluated separately for each accent group. If one accent group has a much higher WER than others, the model is less reliable for speakers of that accent.
Fairness Delta
The WER gap between the best and worst accent groups. A small delta (<10 percentage points) means the model is equitable; a large delta means some accents are under-served.
Why Fairness Matters
A model that works well on standard American English but poorly on other accents would fail for global deployment, accessibility tools, or any diverse user base.

Test 1.7 — Long-Form Audio

Hallucination, repetition, and accuracy on long audio ?
Understanding This Test

Most STT benchmarks use short clips (5–30 s). This test checks what happens with recordings lasting 40–90 seconds or more — closer to real meetings, lectures, and podcasts.

Hallucination
When the model outputs words or phrases that were never spoken. Measured by checking for 5-word sequences in the transcript that don't appear anywhere in the reference text.
Repetition
When the model gets stuck in a loop, repeating the same phrase over and over — a known failure mode in neural sequence models.
Concatenated Audio
Multiple short clips joined end-to-end to create longer files. Tests whether the model handles extended context or degrades after a certain length.
CTC (Connectionist Temporal Classification)
The decoding method used by Parakeet. CTC models process audio frame-by-frame without explicit attention, which can limit their effective context window on very long audio.

If the model degrades past ~40 seconds, production systems should chunk long audio into shorter segments — adding complexity but preserving accuracy.

General-purpose STT models often struggle with specialized vocabulary — company names, ticker symbols, medical terms, or jargon. This test measures that gap.

SPGISpeech

High NE-WER suggests the model would benefit from domain-specific language model rescoring or custom vocabulary boosting for production use in specialized fields.

Test 1.9 — Output Quality

Punctuation, capitalisation, number formatting ?
Understanding This Test

Beyond getting words right, can the model produce "ready to read" text — with proper punctuation, capitalization, and number formatting?

Punctuation F1
A balanced score (0–100%) combining precision (are the inserted punctuation marks correct?) and recall (are the required marks present?). Scored separately for periods, commas, question marks, and exclamation points.
Capitalisation Accuracy
Whether sentence-initial words and proper nouns are correctly uppercased.
Number Formatting
Whether spoken numbers become digits or symbols ("$5 million") rather than being spelled out ("five million dollars").

This matters for downstream use: meeting minutes, captions, and documents all need formatted text. Models that produce raw lowercase text without punctuation require expensive post-processing pipelines.

Test 1.10 — Confidence Calibration

ECE, AUROC, Brier score ?
Understanding This Test

When the model says it's "90% confident" about a word, is it actually correct 90% of the time? This test checks that alignment between confidence and reality.

ECE (Expected Calibration Error)
Measures the average gap between predicted confidence and actual accuracy. A perfectly calibrated model has ECE = 0. Our threshold: ≤ 0.05. Values above 0.1 indicate serious miscalibration.
AUROC (Area Under ROC Curve)
Tests whether confidence scores can separate correct words from incorrect ones. 1.0 = perfect discrimination, 0.5 = no better than random coin flips.
Brier Score
The average squared difference between confidence and outcome (0 or 1). Lower is better. Penalizes both overconfidence and underconfidence.
Reliability Diagram
A chart grouping words by their confidence level and plotting actual accuracy for each group. A well-calibrated model follows the diagonal (50% confidence → 50% accuracy).
"Overconfident"
The model reports high confidence even on wrong words. This makes confidence scores unreliable for automated decisions like "reject if confidence < 0.8."

Test 1.11 — Format Robustness

Audio formats, sample rates, durations, edge cases ?
Understanding This Test

Real-world audio arrives in many different formats and configurations. This test ensures the service handles them all transparently.

Sample Rate
How many audio snapshots are taken per second. Phone calls = 8 kHz, standard speech = 16 kHz, CD quality = 44.1 kHz. The model should resample internally as needed.
Audio Formats
WAV = uncompressed (large files, no quality loss). FLAC = lossless compression (smaller, no loss). MP3 = lossy compression (small, slight loss). OGG = open-source lossy format.
Channels
Mono (1 channel) vs Stereo (2 channels). Most speech is mono, but video recordings are often stereo. The model should handle both.
Edge Cases
Silent audio and white noise test whether the service gracefully returns empty results instead of crashing or hallucinating phantom words.
libvorbis
The codec library needed to decode OGG/Vorbis audio. Its absence on the server causes the OGG failure — a deployment configuration issue, not a model limitation.

Test 2.3 — Prosody Analysis

F0, speaking rate vs LJSpeech reference ?
Understanding This Test

Prosody is the rhythm, stress, and intonation of speech — what makes it sound natural rather than robotic. This test measures pitch and timing patterns.

F0 (Fundamental Frequency)
The pitch of the voice in Hertz. Natural speech has dynamic pitch that rises for questions and falls for statements. Monotone speech has a flat F0 contour.
F0 RMSE
Root-Mean-Square Error between synthesized and reference pitch contours. Lower means more similar pitch patterns, but only meaningful when comparing the same speaker.
F0 Pearson r
Correlation between pitch contours. +1 = identical patterns, 0 = no relationship. Low values are expected when comparing different voices.
WPM (Words Per Minute)
Speaking rate. Normal conversational English ranges from 150–200 WPM. Too fast reduces comprehension; too slow tests patience.
pyworld
A speech analysis library for extracting pitch (F0) from audio using the WORLD vocoder algorithm.

Important caveat: This test compared against LJSpeech (a different female speaker), so pitch differences mostly reflect voice identity rather than prosody quality. A fair comparison needs same-speaker reference audio.

Test 2.4 — Audio Signal Quality

PESQ, STOI, MCD vs reference ?
Understanding This Test

Standard audio engineering metrics that compare two waveforms sample-by-sample. These are the same metrics used by telecom companies to grade phone call quality.

PESQ (Perceptual Evaluation of Speech Quality)
An ITU international standard (P.862) that models how the human ear perceives speech quality. Scale: −0.5 to 4.5. Above 3.0 = acceptable, above 4.0 = excellent.
STOI (Short-Time Objective Intelligibility)
Predicts how much of the speech a human listener would understand. Scale: 0 to 1. Above 0.6 = generally intelligible.
MCD (Mel Cepstral Distortion)
Measures spectral (frequency) difference between two audio signals. Lower is better. Under 8 dB is considered good for TTS.
DTW (Dynamic Time Warping)
An algorithm that aligns two audio signals of different lengths before comparing them, so small timing differences don't dominate the score.

Why results are invalid here: All three metrics require same-speaker reference audio (same voice saying the same words). LJSpeech uses a different speaker than the TTS model (Magpie Aria), so the extreme scores reflect voice identity differences, not synthesis quality.

Test 2.5 — Streaming TTFB & RTF

Latency by text length and interface ?
Understanding This Test

How quickly does the TTS model start producing audio after receiving text? This is the single most important metric for conversational AI and voice assistants.

TTFB (Time to First Byte)
The delay between sending text and receiving the first audio chunk. Under 200 ms feels instantaneous; over 500 ms feels laggy. In voice assistants, this is the pause between the user finishing their question and hearing the first syllable of the answer.
RTF (Real-Time Factor)
Total synthesis time ÷ audio duration. RTF < 1.0 means audio is generated faster than it takes to play — essential for real-time streaming where audio must arrive faster than the listener consumes it.
Text Length Buckets
Requests are grouped by input length (10, 25, 75, 150 words). TTFB should stay roughly constant regardless of text length, since a good streaming system starts outputting audio before processing the full input.

gRPC uses native streaming (audio chunks arrive as generated). REST typically buffers more before responding, hence the ~30 ms TTFB gap.

Test 2.6 — Throughput & Concurrency

RPS, error rates under load ?
Understanding This Test

Simulates multiple users requesting speech synthesis at the same time — finding the server's breaking point before it starts dropping requests.

RPS (Requests Per Second)
Throughput measure — how many synthesis requests complete each second. Ideally scales linearly with concurrency (double the users = double the RPS).
Concurrency Level
The number of simultaneous requests being processed. Tested at 1, 5, 10, and 20 parallel users.
Error Rate
Percentage of requests that fail (timeout, out-of-memory, server overload). Any error rate above 0% in production is concerning and indicates the server is over capacity.
Server Saturation
The point where the GPU runs out of compute or memory. Beyond this, new requests either queue (increasing latency) or fail (increasing error rate). Here, saturation occurs around 10 concurrent requests.

If a voice application needs to serve 100 concurrent users but the server saturates at 10, you'd need 10+ replicas — significantly increasing infrastructure cost.

Test 2.8 — Long-Form Consistency

Voice stability across multi-paragraph synthesis ?
Understanding This Test

When synthesizing a long document paragraph by paragraph, does the voice stay consistent? This matters for audiobooks, email narration, documentation, and IVR systems.

F0 Drift
Variation in average pitch across paragraphs. Large drift (>10 Hz) means the voice sounds noticeably different between sections. Under 8 Hz is considered stable.
WPM Drift
Variation in speaking rate across paragraphs. Drift above 30 WPM would sound like the speaker rushes through some sections and drags through others.
Speaker Similarity (planned)
Uses ECAPA-TDNN neural speaker embeddings to compute a "voice fingerprint" for each paragraph, then checks cosine similarity between them. Not computed here due to missing dependency.
Cosine Similarity
A measure of how alike two voice fingerprints are. 1.0 = identical voice, 0.0 = completely different. Above 0.85 is typically same-speaker.

A 10-paragraph email should sound like one consistent speaker throughout. Voice inconsistency breaks the listening experience and feels unprofessional.
